# Architecture

## Data Flow

1. **Packet source**
   - Live packets arrive from a network interface through libpcap.
   - Offline validation can replay packets from a `.pcap` file.
   - Synthetic mode generates deterministic packet-like events for profiling.

2. **Packet parsing**
   - Ethernet, IPv4, TCP, and UDP headers are parsed into compact `PacketEvent`
     records.
   - Non-IP and non-TCP/UDP packets are ignored in the starter.

3. **Concurrent dispatch**
   - Parsed events enter a bounded queue.
   - The queue provides backpressure and exposes drop counts when the engine
     cannot keep up.

4. **Feature extraction and graph state**
   - Workers update a sharded flow table keyed by source/destination IP, ports,
     and protocol.
   - Each shard owns its own mutex and active flow map, reducing contention.
   - Flow entries expire after an idle timeout to prevent unbounded memory use.

5. **Normalization**
   - C++ loads `feature_config.json` generated by Python.
   - Feature means, standard deviations, feature order, and the anomaly
     threshold are shared between Python and C++.

6. **ONNX inference**
   - Each worker owns an ONNX Runtime session.
   - Workers batch events locally before invoking inference.
   - The exported model returns a reconstruction-error anomaly score.

7. **Alerting**
   - Scores above the saved training threshold are logged as suspicious.
   - This hook is where firewall integration, SIEM forwarding, or rule
     enrichment should be added.

## Why Start With An Autoencoder?

A GNN is the long-term target, but a lightweight autoencoder is a strong first
vertical slice:

- It exports to ONNX reliably.
- It has predictable CPU cost.
- It validates the Python-to-C++ feature contract immediately.
- It lets the C++ ingestion and threading architecture mature before adding
  graph mini-batching complexity.

The C++ `FeatureExtractor` already maintains graph-like flow context. A future
GNN can consume node features, edge indices, and edge attributes emitted from
that state layer.

## Production Hardening Checklist

- Replace synthetic training data with a labeled dataset such as CICIDS or
  UNSW-NB15.
- Add a reproducible feature-generation job for training CSVs.
- Pin the ONNX Runtime version used in CI and deployment.
- Add packet loss and latency histograms, not only counters.
- Tune batch size and worker count under realistic traffic.
- Consider DPDK, AF_XDP, or PF_RING if libpcap cannot keep up.
- Add model versioning and write the model SHA into every alert.
